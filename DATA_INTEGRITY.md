# 数据真实性红线声明

> **版本**: v1.0  
> **生效日期**: 2026-05-16  
> **状态**: 强制执行  

---

## 一、核心原则

**本项目所有涉及金融市场的数据、指标、计算结果，必须 100% 基于真实数据来源。严禁任何形式的编造、臆测、模拟、硬编码或"合理推测"。**

这是一条不可触碰的红线。违反本声明的代码，无论功能多完善，一律不得合入主干。

---

## 二、数据来源白名单

以下是被认可的真实数据来源，任何进入系统的数据必须溯源至其中之一：

| 来源 | 类型 | 用途 | 备注 |
|---|---|---|---|
| **AKShare** | 开源金融数据接口 | 行情、板块、涨停股、北向资金、资金流 | 实时/日终，免费 |
| **Tushare** | 开源金融数据接口（需 Token） | 财务指标、资金流向、基本面数据 | 需配置 `TUSHARE_TOKEN` |
| **上海证券交易所** | 官方公告 | 新股上市、停复牌、公告原文 | 官网/公告接口 |
| **深圳证券交易所** | 官方公告 | 同上 | 官网/公告接口 |
| **北京证券交易所** | 官方公告 | 同上 | 官网/公告接口 |
| **各公司年报/季报** | 官方披露 | 财务数据原始出处 | PDF/公告原文 |

**禁止使用的数据来源：**
- ❌ 任何手写/硬编码的行情或财务数据
- ❌ 随机生成的价格、成交量、财务指标
- ❌ "示例数据"、"演示数据"混入生产代码
- ❌ 未经标注的缓存数据当作实时数据使用
- ❌ 任何无法追溯到上述白名单的"第三方数据"

## 三、行情数据规则

### 3.1 原始行情
- 开盘价、最高价、最低价、收盘价、成交量、成交额 —— 必须直接来自 AKShare/Tushare API 返回。
- 不允许对原始价格做任何形式的修改、平滑、插值、填充。

### 3.2 复权处理
- 复权因子必须来自数据源（AKShare 提供）。
- 若某股票缺失复权因子，该股票在涉及历史比较的因子计算中**直接丢弃**，不能用 `1.0` 或其他默认值顶替。

### 3.3 停牌/退市处理
- 停牌日数据缺失是正常的，**不允许用前一日数据填充**。
- 退市股票在计算截面因子时自然排除（因无当日数据）。

---

## 四、财务数据规则

### 4.1 原始财务指标
- PE_TTM、PB、ROE、毛利率 等 —— 必须来自 Tushare/AKShare 财务报表接口。
- 不允许用价格/市值自行"估算" PE、PB 等衍生指标（除非数据接口只提供原始数据，由我们按公开公式计算，且公式必须审计）。

### 4.2 缺失财务数据
- 若某股票某季度财务数据缺失，**该因子值记为 NaN**，不参与该期打分。
- 严禁用行业均值、历史值、或线性插值填补缺失的财务数据。

---

## 五、因子计算规则

### 5.1 公式可审计
所有因子计算公式必须满足：
- 公式来源公开可查（学术论文、经典教材、行业惯例）。
- 代码中必须保留公式注释，注明出处。
- 任何人拿到同样的原始数据，用同样的公式，应得到同样的结果（允许浮点精度差异）。

### 5.2 已实现的因子公式来源
| 因子 | 公式来源 |
|---|---|
| PE_TTM / PB / PS / PEG | 通行财务定义 |
| EP = 1/PE | Graham 防御型投资 |
| 动量 = (P_t / P_{t-n}) - 1 | Jegadeesh & Titman (1993) |
| 52周新高比 = 当前价 / 52周最高价 | 技术面常识 |
| ROE / ROA / 毛利率 / 净利率 | 通行财务定义 |
| 质量综合因子 = Z-score(ROE+毛利率+营收增长+净利率)/4 | 等权多因子合成 |

### 5.3 未来函数（Look-ahead Bias）禁令
- 任何因子在交易日 T 的计算，**只能使用 T 日及之前已公开的数据**。
- 严禁使用 T+1 才披露的数据去计算 T 日的因子值（例如：用当天收盘后发布的财报去影响当天选股）。

---

## 六、回测规则

### 6.1 数据真实
- 回测必须基于真实历史行情，严禁使用随机游走、蒙特卡洛模拟、或假设分布生成价格序列。

### 6.2 交易费用真实
- 佣金、印花税、过户费、滑点 —— 必须按国内券商真实费率设置。
- 当前默认：佣金万 2.5（最低 5 元）、印花税千 1（卖出）、滑点按实际设置。

### 6.3 信号延迟
- 选股信号在 T 日收盘后产生，**最早在 T+1 日开盘才能成交**。
- 严禁用 T 日收盘价成交 T 日信号（这属于未来函数）。

---

## 七、热点与 NLP 规则

### 7.1 文本来源真实
- 热点抓取必须从真实的财经新闻网站、交易所公告、官方社交媒体获取原文。
- 严禁编造新闻标题或内容。

### 7.2 NLP 只做打分，不生成事实
- NLP 模型仅对已有文本做情感分析、关键词提取。
- 不允许用 LLM "总结"或"改写"后当作事实数据使用。

### 7.3 情感分数透明
- 情感得分必须可解释：给定原文，应能复现分数。
- 使用开源可审计的模型/词典（如 jieba + 自定义情感词典）。

---

## 八、缺失数据处理规范

| 场景 | 处理方式 | 禁止行为 |
|---|---|---|
| 某股票某日行情缺失 | 该股票该日不参与计算 | ❌ 用前值填充 |
| 某股票财务数据缺失 | 该因子记为 NaN | ❌ 用行业均值填充 |
| 某因子历史数据不足（如新股） | 该因子不参与该股票打分 | ❌ 缩短回看周期硬算 |
| 数据源 API 报错/限流 | 记录日志，跳过该批次 | ❌ 用假数据顶替 |

---

## 九、审计与验证机制

### 9.1 代码审计
- 每次 PR 必须通过 `grep` 扫描，确保无 `random`、`mock`、`fake`、`np.random` 等关键词出现在生产代码中。
- 扫描脚本：
  ```bash
  grep -rnE "(random|randn|fake|mock|dummy|simulate|硬编码|假数据|测试数据)" --include="*.py" .
  ```

### 9.2 数据校验
- 落地数据库前，对关键字段做范围校验：
  - 涨跌幅应在 `[-10%, +10%]`（普通股）或 `[-20%, +20%]`（创业板/科创板）
  - 成交量、成交额应为非负数
  - PE_TTM 异常值（如 >1000 或 < -1000）应标记警告

### 9.3 结果可复现
- 同一批原始数据 + 同一套参数，回测结果必须完全一致。
- 所有随机性（如若有）必须通过固定 `random_seed` 消除。

## 十、违规处理

若发现以下行为，立即标记为 **P0 级 Bug**：

1. 代码中存在硬编码的金融数据。
2. 用随机/模拟数据替代真实 API 数据。
3. 因子计算引入未来函数。
4. 用 NLP 生成虚假新闻/热点。
5. 缺失数据未按规定处理（如私自填充）。

处理流程：
1. 发现者立即在 Issue 中标记 `[DATA-REDLINE]`。
2. 相关代码冻结，禁止上线。
3. 修复后需经过双人 Review + 数据溯源验证。
